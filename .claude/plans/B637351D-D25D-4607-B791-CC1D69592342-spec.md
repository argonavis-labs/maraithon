# Spec: Self-serve install flow for Chief of Staff agent

**Ticket:** B637351D — Build self-serve install flow for Chief of Staff agent

## Outcome

A new (non-Kent) user can: land on maraithon.fly.dev → sign in via magic link →
connect at least one connector (Telegram + Gmail/Calendar) → create a project →
install Chief of Staff from a UI button → and receive their first day-brief
within 24 hours. No manual code changes, seeds, or admin API calls required.

## Current-state audit

### What already works (parameterized)
- **Magic-link auth** — `SessionController` (`/auth/magic-link`, `/auth/magic/:token`),
  works for any email, redirects to `/dashboard`. No Kent-specific logic.
- **Install mechanism** — `AgentsLive` `install_library_agent` event →
  `Runtime.install_agent_package/3` → `Agents.install_agent_package/3` creates a
  per-user `Agent` row from the `ai_chief_of_staff` marketplace package and starts
  its runtime process. Fully per-`user_id`.
- **Package definition** — `AgentMarketplace.sync_builtin_packages/0` seeds the
  `ai_chief_of_staff` package from `AgentBuilder.library_specs()` /
  `builtin_manifest/1`: system prompt, skill paths
  (`priv/agents/skills/chief_of_staff/*.md`), `required_connectors`,
  tool/mcp allowlists, `default_config`. Runs on boot.
- **Projects** — `Projects.create_project/2` + create-project form already on
  `DashboardLive`. `Agent` has an optional `project_id` belongs_to.
- **Briefing delivery** — `BriefingCron` polls every 60s,
  `BriefingSchedules.list_due_morning_agents/1` finds `ai_chief_of_staff` agents
  whose local time matches (default 08:00, -5h offset) and that have Telegram
  delivery capability, then schedules the wakeup. First brief fires automatically
  the next morning once the agent exists and Telegram is connected.
- **Connectors** — Google (Gmail+Calendar) and Slack connect via OAuth
  (`/auth/google`, `/auth/slack`); status surfaced on `/connectors` and dashboard.

### What's hardcoded for Kent (must be neutralized for self-serve)
- **Prompts** — `priv/agents/skills/chief_of_staff/morning_briefing.md` ("You are
  Kent's Chief of Staff"); `assistant_harness.ex:291` ("write them for Kent
  directly… use `you` or `Kent`").
- **Org-specific constants** — hardcoded Slack channels (`runner-general`, etc.),
  commercial keywords (`Cogniate`, `Glossier`, …), and team domain `runner.now`
  in `chief_of_staff/acquisition.ex` and `chief_of_staff/skills/morning_briefing.ex`.
- **Auto-install path** — `AgentMarketplace.ensure_default_installations` /
  `default_install_allowed?` are keyed to `Accounts.primary_admin_email()`. This
  is the "Kent gets CoS for free" path; it must not be the self-serve path.
- **Support email** — `kent.fenwick@gmail.com` in `admin_navigation.ex`.

### What's missing
- **Install button is in the wrong place** — it lives on the `/agents` library,
  not the project dashboard, and the ticket asks for it on the project dashboard.
- **No connector gating** — `install_agent_package` always sets
  `install_status: "enabled"` + `runtime_status: "running"` regardless of whether
  required connectors (Telegram, Google) are connected. `install_status:
  "setup_required"` exists in the schema and `Runtime.start_existing_agent`
  already handles it — but nothing sets it. A self-serve install with no Telegram
  connected produces a running agent that can never deliver a brief.
- **No Telegram self-serve connect path** — Google/Slack have OAuth routes;
  Telegram is connected only via the Companion app / device pairing
  (`/companion/auth`). A brand-new web user has no in-app way to link a Telegram
  chat. **This is the critical blocker** — see Open Questions.
- **Install ignores `project_id`** — `AgentsLive` install never passes
  `project_id`, so the agent is not attached to the project the user created.
- **No end-to-end install smoke test** — existing tests
  (`morning_briefing_smoke_test.exs`, `acquisition_test.exs`) cover behavior, not
  the sign-in → connect → install → brief path.
- **No install doc.**

## Scope

### 1. Connector requirements as install-time configuration
- Read `required_connectors` off the `ai_chief_of_staff` package version (already
  populated by `builtin_manifest/1`) and expose them to the install UI: Telegram
  (required — delivery channel), Google/Gmail + Calendar (required for content).
- Add a pre-install readiness check: compare `required_connectors` against the
  user's `ConnectedAccounts`. Surface per-connector connected/missing state.

### 2. "Install Chief of Staff" UI on the project dashboard
- Add an **Install agent** section to `DashboardLive` (project dashboard), with
  Chief of Staff as the single option (row-oriented, Catalyst look per `DESIGN.md`
  — not a marketplace grid).
- The row shows: name, one-line summary, connector requirement chips with
  connected/missing status, and a primary action button.
- Button states:
  - Requirements unmet → **Connect connectors** (links to `/connectors` /
    `/auth/google`, and the Telegram path from Open Q1), button to install
    disabled or labeled "Setup required".
  - Requirements met → **Install Chief of Staff** → installs, attaches the
    current `project_id`, starts the agent.
  - Already installed → show status (running / setup_required) + link to the
    agent inspect panel.
- Install handler: reuse `Runtime.install_agent_package/3`, passing
  `project_id:` and `delivery_policy: %{"telegram" => "enabled"}`. If a required
  connector is missing, install with `install_status: "setup_required"` instead
  of `"enabled"` (do not start a brief-less agent silently).

### 3. Surface prompt + state schema at install time
- The install row links to a read-only "what this agent does" view: system
  prompt summary, the four bundled skills, required connectors, and the default
  schedule (08:00 local brief). Sourced from the package manifest — no new copy.
- Make the timezone / morning-brief hour editable post-install via the existing
  `update_morning_brief_time` event (already on `AgentsLive`) so a non-EST user
  isn't stuck on -5h. Confirm it's reachable from the dashboard or agent panel.

### 4. Neutralize Kent-hardcoding that blocks a generic first brief
- Genericize `morning_briefing.md` and `assistant_harness.ex:291` to address "the
  operator" / "you" instead of "Kent" (use the user's name from `Accounts` if
  available).
- Move the hardcoded Slack channels / commercial keywords / `runner.now` domain
  out of code defaults into per-agent `config` (with the current values as
  Kent's config, not the global default). A fresh tenant should get a working —
  if less tuned — brief with empty org-specific config.
- Repoint the support email or make it configurable.

### 5. End-to-end install smoke test
- New test (e.g. `test/maraithon_web/self_serve_install_smoke_test.exs`) that on
  a fresh tenant: consumes a magic link → stubs Telegram + Google connectors as
  connected → creates a project → installs Chief of Staff via the dashboard
  event → asserts the `Agent` row exists with `behavior: "ai_chief_of_staff"`,
  `install_status: "enabled"`, the correct `project_id`, and that
  `BriefingSchedules.list_due_morning_agents/1` would pick it up at its scheduled
  local time. Also assert the "requirements unmet" branch yields
  `install_status: "setup_required"` and no scheduled brief.

### 6. Install doc
- One-page `docs/install-chief-of-staff.md`: sign in → connect Telegram → connect
  Google → create a project → click Install → expect the first brief by 08:00
  local the next day → where to change the brief time. Written for a non-Kent user.

## Non-goals
- Marketplace UI / multiple installable agents — Chief of Staff only.
- A landing page (separate ticket).
- Visual agent builder.
- Deep per-tenant tuning of Slack channels / keywords beyond moving them into
  config; smart defaults can be a follow-up.

## Open questions / risks
1. **Telegram self-serve connection (blocker).** There is no OAuth-style web flow
   to link a Telegram chat — only the Companion app / `/companion/auth` device
   pairing. Without solving this a non-Kent user cannot receive a brief.
   Options: (a) a "connect Telegram" flow using a bot deep-link + chat-id capture,
   (b) require the Companion app and document it, (c) allow email/other delivery
   for the first brief. **Decision needed before implementation.**
2. Should install be blocked until connectors are ready, or allowed as
   `setup_required` with a clear resume path? Spec assumes the latter.
3. Does `ensure_default_installations` need a guard so self-serve users don't
   collide with the primary-admin auto-install path? Likely yes — confirm.
4. Default timezone is -5h (EST). Acceptable for v1 with post-install editing, or
   should the install UI ask for timezone up front?

## Acceptance criteria
- A fresh user, with no code changes, can complete sign-in → connect → create
  project → install from the dashboard button → and the installed agent is
  scheduled for a brief within 24h.
- Install is blocked or marked `setup_required` when required connectors are
  missing; never produces a silently brief-less "running" agent.
- The first brief contains no "Kent"-specific copy or org-specific constants
  baked into code.
- The end-to-end smoke test passes; `mix precommit` is clean.
- `docs/install-chief-of-staff.md` exists and a non-Kent user could follow it.
