# Spec: Run 30-day Chief of Staff dogfood with crash-recovery telemetry

**Ticket:** EFAA7D26 — Run 30-day Chief of Staff dogfood with crash-recovery telemetry

## Outcome

A 30-day continuous production run of the Chief of Staff agent on Kent's
accounts (single tenant). At the end: a written report covering total uptime,
every crash + recovery (cause + outcome), in-flight work that survived
restarts, and any user-visible regressions — plus one demo-able
crash-and-recover clip/transcript suitable for a landing page or alpha pitch.

This is a **stability + observability** run. The deliverables are telemetry,
a daily digest, and a report — not new agent behavior.

## Current state (what exists, what's missing)

Verified in the codebase:

- **The agent:** `Maraithon.Behaviors.AIChiefOfStaff` runs inside a
  `Maraithon.Runtime.Agent` (`gen_statem`), supervised by the
  `Maraithon.Runtime.AgentSupervisor` `DynamicSupervisor`. The agent child spec
  is **`restart: :temporary`** (`lib/maraithon/runtime/agent.ex:63`) — a
  crashed agent is **not** auto-restarted by its supervisor.
- **App-restart recovery works:** `Maraithon.Runtime.Bootstrap` runs on boot
  and calls `Runtime.resume_all_agents/0`, which restarts every agent whose DB
  status is `running`/`degraded`. So a node restart (Fly redeploy, OOM) does
  recover agents; a lone agent-process crash mid-cycle does not.
- **In-flight work is already durable:** the `effects` outbox
  (`Maraithon.Effects`, status + `attempts`/`max_attempts`/`retry_after`) and
  the `scheduled_jobs` table (`Maraithon.Runtime.Scheduler`) survive restarts
  and are re-driven by `EffectRunner` / `Scheduler`. Per-run history lives in
  `agent_runs` / `agent_run_steps`.
- **Event log exists but is per-agent:** `Maraithon.Events` (`events` table)
  records `agent_started`, `effect_failed`, `checkpoint_created`,
  `agent_stopped`, etc. It requires an `agent_id`, so it cannot hold
  node-level or supervisor-level incidents.
- **Health is logged, not persisted:** `Maraithon.Runtime.HealthReporter`
  logs `Maraithon.Health.check/0` every 60s (status, agents running/degraded,
  memory, uptime). Nothing is stored or summarized.
- **No state replay:** `agent.ex:119` and the checkpoint stub (`agent.ex:~500`)
  are TODOs — recovered agents start fresh, not from a snapshot. **In scope to
  observe, out of scope to build** (see Non-goals).
- **Telegram send path:** `Maraithon.Connectors.Telegram.send_message/3`, with
  the chat id from `Maraithon.ConnectedAccounts.telegram_destination(user_id)`.
  `Maraithon.Runtime.BriefNotifier` and `Maraithon.Runtime.BriefingCron` are
  the existing patterns for scheduled Telegram pushes.
- **Deployment:** Mix releases via `Dockerfile` / `entrypoint.sh` / `fly.toml`;
  migrations via `Maraithon.Release.migrate/0`. (An `erl_crash.dump` sits in the
  repo root — prior crashes have happened.)

**Gap:** there is no single, queryable record of "the runtime went down / an
agent crashed / it came back", and no daily rollup. That is what this ticket
builds.

## Scope

### 1. Runtime incident log (structured crash-recovery telemetry)

Add `Maraithon.Runtime.IncidentLog` backed by a new `runtime_incidents` table:

| column | notes |
|---|---|
| `id` | binary_id |
| `kind` | `node_boot`, `node_shutdown`, `agent_crash`, `agent_resumed`, `agent_stopped_unexpectedly`, `db_outage`, `db_recovered` |
| `agent_id` | nullable (node-level incidents have none) |
| `reason` | crash reason / exit signal / cause string |
| `metadata` | map — uptime at event, memory, restart count, behavior, last sequence_num, etc. |
| `node` | node name |
| `occurred_at` | utc |

Wire it at these hook points (instrument existing code; do not duplicate data
already in `effects` / `agent_runs`):

- **Node boot / shutdown:** emit `node_boot` from `Maraithon.Runtime.Bootstrap`
  (it already runs once on startup); emit `node_shutdown` from an
  `Application` stop hook or a dedicated terminating GenServer.
- **Agent crash:** add a lightweight `Maraithon.Runtime.AgentWatcher`
  GenServer that `Process.monitor/1`s each running agent (subscribing as
  agents start/stop) and records `agent_crash` with the exit reason when a
  `:DOWN` arrives with a non-`:normal` reason. Because agents are
  `:temporary`, this watcher is the only thing that will notice.
- **Agent resume:** emit `agent_resumed` from `Runtime.resume_all_agents/0` /
  `start_agent_process/1`, tagging whether it followed a node boot or a
  targeted re-resume.
- **DB outages:** emit `db_outage` / `db_recovered` from the failure-marking
  path in `Maraithon.Health` (it already tracks a failure timestamp in
  `:persistent_term`).

### 2. Minimal agent recovery monitor

Because agent processes are `:temporary`, a mid-cycle crash currently leaves
the agent dead until the next node restart — which would make a 30-day
"crash-recovery" run mostly measure nothing. Add a **minimal** recovery action
to `AgentWatcher`: on an abnormal `:DOWN`, after recording `agent_crash`,
re-resume that agent via the existing `Runtime` resume path (with a small
backoff and a crash-loop circuit breaker — e.g. stop re-resuming after N
crashes in M minutes and record `agent_stopped_unexpectedly`).

This is recovery plumbing built entirely on existing resume code — it adds **no
new agent capability** and no snapshot/replay. Flag for confirmation (see Open
questions) since it is the one behavior change in an otherwise observe-only
ticket.

### 3. Daily uptime / crash-recovery digest via Telegram

Add `Maraithon.Runtime.DogfoodDigest` (GenServer, same shape as
`BriefNotifier`/`BriefingCron`): once per day at a configured local time,
compose and send a Telegram message to Kent covering the trailing 24h:

- uptime % and longest continuous uptime streak
- count of each incident `kind`, with one line per `agent_crash` (cause +
  whether re-resume succeeded)
- in-flight work at the last restart vs. after: pending/failed `effects`,
  pending `scheduled_jobs`, `agent_runs` left in `running` then resolved
- current `Maraithon.Health.check/0` snapshot (status, memory, agent counts)

Send via `Maraithon.Connectors.Telegram.send_message/3` to
`ConnectedAccounts.telegram_destination/1`. All values come from
`runtime_incidents`, `effects`, `scheduled_jobs`, `agent_runs`, and `Health` —
no new data sources.

### 4. The 30-day run

- Run on Kent's production deployment for 30 calendar days.
- No manual interventions other than logging issues — if the recovery monitor
  cannot bring an agent back, that is a finding, not something to hand-fix.
- Daily digest is the running record; spot-check it each morning.
- Capture at least one real production crash-and-recover as a recording or
  transcript (the digest line + the Telegram thread, or a screen capture of
  the agent resuming and completing previously in-flight work).

### 5. Wrap-up report

At day 30, write `docs/dogfood/2026-chief-of-staff-30-day.md` (or similar):

- total uptime, incident counts by kind, MTBF / MTTR
- every crash: cause, recovery outcome, in-flight work lost vs. preserved
- user-visible regressions observed during the run
- what worked, what broke, and the explicit list of what the runtime still
  needs before external users (e.g. snapshot/replay, agent supervisor restart
  strategy, crash-loop handling)
- link to the demo clip/transcript

## Non-goals

- **No new agent capabilities or skills** — observability + recovery plumbing only.
- **No snapshot / event-replay system** — the `agent.ex:119` and checkpoint
  TODOs stay TODOs. The run measures the cost of *not* having them; the report
  recommends whether to build them.
- **No multi-tenant / multi-user run** — single tenant, Kent's accounts.
- **No new dashboards/UI** — the Telegram digest is the surface. (A read-only
  admin view of `runtime_incidents` is a possible follow-up, not this ticket.)
- **No external alerting integrations** (PagerDuty, etc.).

## Open questions / risks

1. **Recovery monitor (Scope 2) — confirm before building.** It is the only
   behavior change. Alternative: leave agents `:temporary` and let the run
   simply document every unrecovered crash. Recommendation: build the minimal
   monitor, otherwise the run under-measures recovery.
2. **Crash-loop protection thresholds** (N crashes / M minutes) need values.
3. **Digest delivery time** — needs a configured local hour for Kent.
4. **Day-0 baseline** — capture current `effects`/`agent_runs`/`scheduled_jobs`
   backlog so digest deltas are meaningful.
5. **"Survived restarts" definition** — proposal: an effect/job/run counts as
   "survived" if it was non-terminal before a restart and reached a terminal
   state after, attributable to the same logical work item.
6. **Stale `erl_crash.dump` in repo root** — investigate and `.gitignore`
   before the run so new dumps are noticed.

## Acceptance criteria

- [ ] `runtime_incidents` table + `Maraithon.Runtime.IncidentLog` context, with
      migration and tests.
- [ ] Incidents recorded for node boot/shutdown, agent crash, agent resume, and
      DB outage/recovery — verified by an integration test that crashes an
      agent and asserts an `agent_crash` + `agent_resumed` pair.
- [ ] `AgentWatcher` re-resumes an abnormally-crashed agent and trips its
      circuit breaker after the configured crash-loop threshold.
- [ ] `DogfoodDigest` sends a correct daily Telegram summary; covered by a test
      with a fake Telegram client.
- [ ] All new config keys documented and defaulted; `mix precommit` clean.
- [ ] 30-day run completed with daily digests archived.
- [ ] At least one real crash-and-recover captured as clip/transcript.
- [ ] Wrap-up report committed under `docs/dogfood/`.
